Bab 1 — Pendahuluan

Latar Belakang

Sebagian besar showroom kendaraan di Indonesia — terutama skala menengah — masih mengandalkan pencatatan manual dan proses pemesanan via telepon/pertemuan langsung. Setidaknya ada 4 titik kegagalan utama:

❌

Ketidakakuratan Data Inventaris

Status ketersediaan unit tidak real-time → pelanggan memesan unit yang sudah terjual.

⚠️

Minimnya Transparansi Transaksi

Pelanggan tidak bisa verifikasi pembayaran secara mandiri — harus hubungi staf aktif.

🐌

Proses Verifikasi yang Lambat

Kasir periksa bukti transfer manual satu per satu — rawan kesalahan saat volume tinggi.

💾

Risiko Duplikasi & Kehilangan Data

Tanpa basis data ternormalisasi, riwayat transaksi rentan terduplikasi atau hilang.

Solusi

Platform Web Terintegrasi

ShowDrive memindahkan seluruh alur transaksi showroom — dari penemuan unit di katalog, pengajuan pemesanan, unggah bukti transfer, hingga penerbitan kwitansi digital — ke dalam satu platform yang dapat diakses kapan saja dan dari mana saja.

Tech Stack

Backend Framework Laravel 12 / PHP 8.2
Database MySQL 8.0 (InnoDB)
Frontend Blade + Tailwind CSS
Async Operations Fetch API / AJAX

Bab 3.5 — Entity Relationship Diagram

Perancangan Database

6 Entitas Utama (Normalisasi 3NF)

companies

id, name, tax_id, phone, bank_name, bank_account, qris_image, logo_url

warehouses

id, company_id→companies, name, location

cashiers

id, company_id→companies, name, username, password(bcrypt)

items

id, warehouse_id, brand, model, vin[UQ], year, price, dp_percentage, status[ENUM], engine, transmission, color, image_url

customers

id, name, phone[UQ], nik[UQ,NULLABLE]

invoices ★

id, invoice_code[UQ], customer_id, item_id, cashier_id[NULL], date[DATETIME], subtotal, tax_rate(11%), tax_amount, total_amount, paid_amount, payment_type, payment_status, status, authentic_receipt, rejection_note, cancellation_note, handed_over_at

3 Kebijakan ON DELETE Berbeda

RESTRICT

items → invoices

Tolak penghapusan unit kendaraan jika masih ada invoice aktif. Proteksi aset finansial paling kritis.

CASCADE

customers → invoices

Hapus pelanggan → seluruh invoice ikut terhapus atomik. Cegah data orphan di tabel invoices.

SET NULL

cashiers → invoices

Hapus akun kasir → cashier_id di-set NULL. Histori transaksi tetap utuh meski personel berganti.

Siklus Status Unit

Available → booking → Invoiced → lunas → Sold ★

★ Status Sold bersifat permanen setelah handover dikonfirmasi kasir.

Bab 2.5 — Keunggulan Sistem

Fitur Unggulan Teknis

CORE

⚔️

Proteksi Race Condition

Dua pelanggan booking unit yang sama bersamaan → hanya satu yang berhasil.

DB::transaction()
 → Item::lockForUpdate()->find($id)
 → check status === 'Available'
 → create Invoice + update status

🔐

Sistem OTP Berlapis

Pelanggan akses riwayat tansaksi tanpa akun/password — via OTP nomor HP.

  • ▸ random_int(1000,9999) → di-hash Bcrypt
  • ▸ Expire 5 menit · maks 3× percobaan
  • ▸ Verifikasi via Hash::check() — bukan string compare
  • ▸ Rate limiter 5×/10 menit per IP

🔢

Kode Invoice Anti-Duplikasi

Format SD/2026/0001 — sequential numbering aman dari race condition.

lockForUpdate() → ambil nomor terakhir
→ increment + format SD/YYYY/NNNN
→ commit sebelum request ke-2 baca nilai

🛡️

4 Lapisan Keamanan

Bekerja berlapis dari UI hingga database level.

1 Security by obscurity — URL login tersembunyi
2 Rate limiting per IP (OTP / login / booking)
3 Session-based OTP dengan Bcrypt hash
4 Database-level integrity (ON DELETE constraints)

Bab 6.2 — Saran Pengembangan

Saran Pengembangan

💬

Notifikasi Otomatis via WhatsApp API

Prioritas Tinggi

Saat ini notifikasi WA menggunakan prefilled link yang diklik kasir secara manual. Integrasikan WhatsApp Business API (Fonnte / WaBlas) agar notifikasi perubahan status terkirim otomatis tanpa intervensi kasir.

💳

Integrasi Payment Gateway Otomatis

Prioritas Tinggi

Otomatisasi verifikasi pembayaran dengan Midtrans / Xendit. Status invoice diperbarui otomatis saat webhook konfirmasi pembayaran diterima — mengeliminasi pemeriksaan bukti transfer manual oleh kasir.

🧪

Automated Testing (PHPUnit)

Kualitas Kode

Tambahkan Feature Test menggunakan PHPUnit (bawaan Laravel) untuk skenario kritis: booking atomik, verifikasi OTP, dan penolakan hapus unit bertransaksi. Meningkatkan regression safety saat pengembangan fitur baru.

🏢

Dukungan Multi-Cabang Showroom

Skalabilitas

Arsitektur database ShowDrive sudah memisahkan entitas Company & Warehouse — fondasi multi-cabang sudah tersedia. Panel manajemen multi-cabang dapat ditambahkan tanpa perombakan skema database yang signifikan.

📱

Optimasi Mobile-First

Refaktor UI untuk pengalaman smartphone — mayoritas pengguna internet Indonesia via mobile.

📦

Multi-Unit per Transaksi

Saat ini 1 unit per invoice. Dukung transaksi multi-unit untuk pembelian armada korporat.

⚙️

Redis Queue Production

Konfigurasi Redis untuk queue driver agar penghapusan file lama tidak memblokir respons HTTP.

Bab 6 — Penutup

Terima Kasih

Kesimpulan

ShowDrive berhasil mewujudkan seluruh 6 tujuan pengembangan yang dirumuskan — dari digitalisasi inventaris, alur pemesanan mandiri pelanggan, keamanan OTP berlapis, validasi AJAX real-time, konsistensi data ACID, hingga laporan keuangan dengan ekspor CSV.

Kontribusi

Sistem ini membuktikan bahwa arsitektur Three-Tier Laravel + MySQL mampu menopang seluruh siklus bisnis showroom dari pemesanan hingga serah terima unit, dengan keamanan dan konsistensi data yang terjamin di level basis data.

Potensi ke Depan

Dengan penerapan saran pengembangan secara bertahap — integrasi WhatsApp API, payment gateway, dan dukungan multi-cabang — ShowDrive berpotensi berkembang menjadi platform penjualan kendaraan premium yang komprehensif dan siap produksi.
